# Knowledge Admin dashboard — V1 surfaces design?

## Status

**Open** — awaiting team decision.

## Why it matters

The [[knowledge-admin|KM Admin]] is the **buyer**. Their V1 dashboard is what justifies the [[consilium-business-model|practice-group license price]] from their seat. The design-guidelines doc explicitly excludes admin dashboard from MVP design scope, but V1 ships it without a designed spec. (see [[knowledge-admin]], [[consilium-design-guidelines]] §12)

## What we know so far

V1 admin power per [[consilium-product-requirements]] §4.1:

- Approval queue (pending count badge)
- Side-by-side preview of original (with privilege markers) and redacted versions
- Approve / reject / edit-and-approve actions
- Edit and remove team-graph nodes
- Manage team membership
- View audit logs

What is *not* designed: layout, information density, default sort order, filter chips, escalation rules.

## What would resolve it

- A V1 wireframe for the four main admin surfaces: approval queue, team-graph editor, audit-log query, analytics.
- Decide whether analytics live in the same dashboard or are a separate "KM Analytics" surface.
